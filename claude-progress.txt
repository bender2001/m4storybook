    variant that silently fails. Combine

    (e.g. `bg-surface-variant`) for a token-driven disabled

- Re-usable lessons from this slice:
  * Tailwind utility cascade is unpredictable for "later one wins"
    rules involving multiple color utilities (text-error vs
    text-on-surface-variant on the same element). Tailwind sorts
    utilities by category and color order from the config — so the
    color whose name appears later in the colors[] array wins,
    regardless of source-order in the className string. Fix: pick a
    single color in JS and pass exactly one text-* class to the
    element. Same trick used in the Switch errorClasses table.
  * Same lesson for the field's border-color: even though
    [&[data-error=true]]:border-error appeared LAST in the variant
    string, [&[data-focused=true]]:border-primary kept winning. The
    safe pattern is the Select / Switch escape hatch: keep the
    arbitrary-variant rules AND append a JS-computed `error &&
    "border-error"` so an explicit class always wins.
  * Floating-label click-through is awkward for Playwright. The label
    is positioned absolute over the input region; whether you keep
    pointer-events:none (input intercepts the click) or remove it
    (label intercepts the click), Playwright's actionability check
    fails. Workaround: test the M3 UX directly — clicking the field
    gutter focuses the input via the field's onClick handler.
  * data-* attributes on a NESTED div need their own data-variant /
    data-error / data-size copy; arbitrary-variant Tailwind rules
    target the element they're on, not the root. Putting the prop
    once on the root and assuming descendants inherit will silently
    lose styles.
